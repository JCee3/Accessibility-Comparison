Introduction

This page will take you through some activities for each section of the website. These activities are designed to help you understand how it feels to need certain accessibility requirements that haven't been met. You may wish to keep this page open in a separate window for easy reference. You will need to use the Google Chrome web browser for certain activities.

Each section of this website has an "accessible" and an "inaccessible" version, and this page is no exception. However, there are no activities relating specifically to this page as it introduces no new functionality.

Homepage

Keyboard-only Navigation

  1. Try navigating through the inaccessible homepage using only your Tab and Arrow keys.
  2. Note how none of the options on the navigation bar are highlighted, and nor is the "Make me accessible" "link". This is because they're not buttons, but rather <span> elements with "onclick", which are not natively added to the tab order like buttons are. The "make me accessible" is actually an image of text, which has the same problem as spans with the added issue of the text being impossible for a screenreader to parse.
  3. More information on these issues can be found in the Vision and Dexterity sections.
  4. It's much easier to navigate through the accessible homepage using Tab and Arrow keys, because semantic elements are being used which allow keyboard navigation by default.

Vision

Activity 1 - Alt Text

  1. On the "Inaccessible Vision" page, read the "Alt text and descriptions of video content" section and follow the instruction in it. Notice how the content of the alt text actively makes the activity more difficult, as the entire contents will be read to a screenreader user.
  2. Repeat the activity on the "Accessible Vision" page, which should be much easier.

Activity 2 - Colourblindness

  1. First, download the Colorblindly Chrome extension. If you have trouble getting it to work, the FAQs has a troubleshooting section.
  2. Open the "Inaccessible Vision" page and set the Colorblindly extension to use the "Green-Blind/Deuteranopia" setting.
  3. Read the "Sensory characteristics of UI" section and follow the instruction in it.
  4. Repeat this for the "Accessible Vision" page. Note how even though the colours are still just as difficult to distinguish between, it is much easier to complete the task.
  5. You may wish to turn off Colorblindly for subsequent Vision activities.

Activity 3 - Responsive Design

  1. This activity can be done on any section of the website, but is most relevant to Vision which has information on this requirement.
  2. Go to an inaccessible page (e.g. "Inaccessible Vision") and inspect it using Chrome's Developer Tools (right-click > Inspect).
  3. Toggle the Device Toolbar, set Dimensions to "Responsive", and set the width of the viewport to 320.
  4. 320 CSS pixels is the minimum width at which all content must be accessible without horizontal scrolling. This benefits users who require a very high zoom level to access content, as well as conforming to standard responsive design for mobile devices.
  5. Note that much of the navigation bar becomes unreachable, extending out past the edge of the viewport.
  6. Repeat this task on an accessible page (e.g. "Accessible Vision").
  7. See that the presentation of the navbar changes. It is now a vertical stack unhidden and hidden by a single button. No content or functionality is lost, despite the much narrower screen.
  8. Note also that at "desktop" dimensions the video on the "Inaccessible Hearing" page expands to fill the whole screen because no max-width was set for larger viewports. Filling the screen by default is not desired behaviour in this case, and there is already a button for fullscreen video.

Hearing

Captions

  1. On the "Inaccessible Hearing" page, watch the video. Consider how difficult it is to determine what's being said - is Jessica saying she's faking deafness?
  2. Go to the "Accessible Hearing" page, and watch the video. Notice how you now know exactly what's being said, and how an impression completely opposite to the truth was given in the non-captioned video.

Dexterity

Activity 1 - Skip Content

  1. Go to the "Inaccessible Dexterity" page. Try to navigate down to the text content using the Tab key.
  2. Notice how many interactive sections there are, that they may be things the user absolutely does not want to interact with, and how long it takes to get to the desired location. Consider how your experience would be impacted if this was the case for every webpage you visited.
  3. Go to the "Accessible Dexterity" page. Press the Tab key once.
  4. Observe the "Skip to content" link which has appeared in the top left corner of the viewport, and which will disappear when it does not receive focus.
  5. Press the Enter key and see that the header/navigation bar has been skipped and the unique page content is now displayed.

Activity 2 - Modal

  1. On the "Inaccessible Dexterity" page, use the keyboard to navigate to and interact with the "Open Modal" button.
  2. Try pressing the Escape button and see if it closes the modal.
  3. Can you navigate to the Close, Cancel or Confirm buttons using the keyboard? Try pressing Tab a few times. Notice that input isn't confined to only affect the modal content as it should, but rather affects the page behind the modal.
  4. It is impossible to close the modal without the use of the mouse. This is one example of a keyboard trap.
  5. Repeat this activity on the "Accessible Dexterity" page. Note that tabbing only selects the three buttons on the modal and the URL bar in your browser, and using the Escape key closes the modal.
