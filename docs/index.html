In February 2024, only 4.1% of the top million web home pages were digitally accessible, based on automated accessibility checking.

This checking only covers 25-30% of accessibility requirements.

Introduction

The Web Content Accessibility Guidelines (WCAG 2.2) provide extensive and valuable guidance on the requirements to make the web accessible. There are three levels of compliance (A, AA, and AAA), with Level A being the absolute minimum required. This page aims to inform readers from a developer perspective of the basics needed to make their pages accessible to users with a wide range of disabilities and access requirements. Additionally, it is intended to help users understand a little of how it feels to have accessibility requirements that aren't met.

Please see the Instructions section for activities to carry out.

Below are some requirements that apply to multiple disabilities, or which do not fit into the categories above - more specific requirements are described in the pages for each disability.

General Requirements

Information and Relationships, Meaningful Sequence

  • Use semantic elements - heading (<h>) tags, navs for navigation, link (<a>) and button elements rather than divs or spans with onClick().
  • Never use the CSS content property to provide meaningful text - this will not be included in the DOM and is thus inaccessible to anyone using tools that rely on the DOM.
  • Content should have a logical order, and the focus order should follow the reading order.
    • For example, in a navigation bar, as above, tabbing through should focus on "Homepage", "Vision", "Hearing", "Dexterity", "FAQs" and "Instructions", in that order. Jumping around (e.g. starting with "Dexterity") can confuse users and make the page harder to use.
  • Content should behave in the way it's shown - if something looks like a hyperlink it should change the page the user is on, not do something on the page like a button.
    • It's acceptable to remove link underlines in situations such as navbars, as long as it's made clear that they're interactive.
  • It should be obvious to the user where they are in the website. Note how the current page is bolded and underlined in the navbar above. Another example of this is the use of breadcrumb components for visibility of the sequence followed to navigate to a page, particularly useful in ecommerce with nested products and categories.

Seizures and physical reactions

  • Flashing lights can cause physical reactions in people with certain health issues, whether those are seizures, migraines, or another response.
  • The minimum requirement to prevent these reactions is that nothing on the page should flash more than three times in one second, or the flash should be below general flash and red flash thresholds. The latter of these can be quite complex to assess and may require the use of a tool.
  • If a video on the page breaches this, provide a seizure warning, as well as a text alternative to the video content.
  • This website does not showcase failure to meet this requirement due to the safety concerns that would cause.

Parsing

In content implemented with markup languages, elements:

  • Should have complete start and end tags.
    • Modern browsers have repair techniques, which allow them to parse broken HTML, but assistive technologies like screenreaders may not.
    • An incomplete heading element's text may be accessed by a screenreader, but not be added to the list of headings.
  • Are nested according to their specifications.
  • Don't contain duplicate attributes.
  • With the id attribute have unique IDs.
    • Not using unique IDs means that screenreaders cannot accurately access any instance of an element with that ID other than the first.
Accessibility isn't a feature. Lack of accessibility is a bug.
