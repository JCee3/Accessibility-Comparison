Context

People may have limited dexterity for many different reasons, such as:

  • Paralysis from brain, spinal or nerve injury, including from such causes as stroke or quadriplegia.
  • Amputation or congenital difference meaning the user does not have ten fingers, two hands/arms, etc.
  • Poor control of hands or fingers due to tremors, weakness or spasticity, which could be caused by conditions such as cerebral palsy, muscular dystrophy, or multiple sclerosis.
  • Extreme fatigue, whether caused by illness (e.g. M.E., heart disease) or not.
  • Temporary disability, such as from a broken limb or repetitive strain injury.
  • Situation, such as needing to hold something with one hand/arm (e.g. a child, food) and only being able to freely use the other.

While some users may control their devices with their voice, not every user in this category will. It's especially important to consider keyboard-only users.

Requirements

Keyboard use

Operable only with keyboard

Try to access the following button using only the keyboard (Tab, arrow, and Enter keys):

  • All content functionality should be operable through a keyboard interface, without requiring specific timing for individual keystrokes.
    • The only exception to this is when the underlying function requires input that depends on the path of the user's movement and not just the endpoints.
    • Please be aware of the difference between function and input technique. If handwriting is being used to enter text, the input technique requires path-dependent input, but the underlying function of inputting text does not. In this case, the user should be able to choose to enter text through the keyboard, not just a touchscreen or other pad.
    • If drag-and-drop/swiping is an important input mechanic for the page, an alternative must be available and obvious. Often a button is the most suitable substitute.
  • This is one of many occasions where using appropriate semantic elements will make it easier to create accessible sites, as using the correct element for the situation means elements that should be focussable generally will be. If an element needs to become focussable, adding tabindex=0 will at least place it inside the tab order.

Keyboard traps

  • Be aware of keyboard navigation when constructing forms. Radio button groups should be grouped using fieldset, and individual entries should be focussable using arrow buttons, with tab taking the user out of the fieldset to the next group.
  • Additionally, be very conscious of any use of the autofocus attribute for an input field - it can be extremely confusing for users to start in what could be the middle of the page.

How familiar are you with digital accessibility?
  • Keyboard traps, components which can have focus moved to them using the keyboard but cannot have focus moved away from them using the keyboard, must be avoided.
    • If the keys needed to move away from a component are more than unmodified arrow/tab keys, the user should be advised of this.
    • Modals are a common cause of keyboard traps - ensure there is a way to exit your modals that doesn't require the use of a mouse. The HTML <dialog> element should be used for modals whenever possible, as it handles keyboard-only use appropriately.
  • Modals should always have a semantic h1 heading (this may be styled to look smaller) and a body. The heading should indicate the content of the modal and/or reason for its existence.

Skip content blocks

  • It should be possible to skip past blocks of content that is repeated on every page (e.g. navigation, ad frames). This is often implemented as a "skip to content" link placed at the top of the page.
  • These links can still be necessary even in smaller viewports - people using voice control or screenreaders on mobile devices, or those with high zoom levels on larger devices, still benefit from being able to skip repeated content.
  • Entirely visually hiding these links for the sake of aesthetic prevents sighted keyboard users, those most likely to not have additional navigation methods, from knowing they exist and thus being able to use them.
  • However, a permanent link doesn't always fit the planned user interface design. When it comes to "desktop" use, the best compromise between these conflicting needs is to have a skip to content link initially placed off-screen (not with display:none as this hides the link from use entirely) which then is moved very obviously into view when it receives keyboard focus. This is done on every accessible page of this website other than the homepage for demonstration purposes.

Mouse or mobile use

Consistent placement

  • It's important that elements don't jump around the page, particularly if this happens during use and is not in response to user action.
  • A button moving out from underneath a finger tap or cursor, leading to a misclick/mistap, is incredibly frustrating for all users, but particularly those with tremors or who otherwise need to expend more effort on motor planning.
